# Type Design Analysis — `soph/more-improvements-fixes`

## Type: `oauthhttp.OAuthErrorResponse`

File: `/Users/alex/workspace/auth-go/internal/oauthhttp/error.go:14-17`

### Invariants Identified
- `Error` is the OAuth-defined error code (RFC 6749 §5.2 token strings).
- `ErrorDescription` is server-supplied free text that is **explicitly unsanitised** — callers must run it through `SanitizeDescription` before logging/printing.
- Returned non-nil only when `Error` is non-empty after `TrimSpace` (enforced inside `ReadOAuthError`, `error.go:33-34`).

### Ratings
- **Encapsulation**: 4/10. Both fields are exported and mutable; nothing stops a caller from constructing or mutating an instance with control characters in `Error`. The only construction path inside this package goes through `ReadOAuthError`, but the type itself doesn't enforce that.
- **Invariant Expression**: 3/10. The "ErrorDescription is unsanitised, you must sanitise before printing" rule lives in doc comments only (`error.go:20-24`). Two callers (`deviceflow.go:487`, `sts/sts.go:433`) get this right today, but the type has no way to express "this is a tainted string". The shared `string` shape between `Error` (well-defined small enum-ish set) and `ErrorDescription` (hostile-controlled free text) is exactly the case typed string aliases handle well.
- **Invariant Usefulness**: 7/10. The shape mirrors the wire and is what callers want. The "trim error code, return nil if empty" rule in `ReadOAuthError` is a useful guard against a server returning `{"error":"  "}`.
- **Invariant Enforcement**: 5/10. Construction is funneled through `ReadOAuthError`, but the struct is exported with public fields — any package importing `oauthhttp` could materialize one bypassing the constructor. The sanitisation requirement is **not** enforced on the type; it lives in documentation only, and one stale call site can leak ANSI escape sequences into terminals (which is the bug class this PR is otherwise trying to close).

### Strengths
- Bounded body read via `readLimitedBody` (`error.go:48-57`) hardens the constructor against memory exhaustion.
- Empty-error fallback (`error.go:36-38`) handles real-world auth servers that return `{"error":"","error_description":"..."}` shapes without surfacing a confusing empty-code error.

### Concerns
1. **The "unsanitised" invariant is documentation-only.** This is the highest-risk concern given that the entire PR is about hardening OAuth error handling. If someone writes a third call site and forgets `SanitizeDescription`, terminal escape injection is back. A type that documents "callers must sanitise" is exactly the anti-pattern flagged in the rubric.
2. **`Error` field name shadows the `error` interface concept.** Calling `apiErr.Error()` would be confusing if the type ever grew an `Error()` method (e.g. to make it satisfy `error`). Not blocking, but a smell.
3. **No `Error()` method.** The type carries error semantics but isn't an `error`. Callers wrap it manually via `fmt.Errorf` at each site (`sts.go:433`, `deviceflow.go:487`) and reproduce the "status N: desc" template. Drift hazard.

### Recommended Improvements
- Make `ErrorDescription` a typed alias to bake the taint into the type system:
  ```go
  type UnsafeDescription string  // unsanitised; pass through SanitizeDescription
  type OAuthErrorResponse struct {
      Error            string
      ErrorDescription UnsafeDescription `json:"error_description"`
  }
  func (d UnsafeDescription) Sanitized() string { return SanitizeDescription(string(d)) }
  ```
  Callers then write `apiErr.ErrorDescription.Sanitized()` and the requirement is impossible to miss. Low complexity, high payoff.
- Consider giving `OAuthErrorResponse` an `Error() string` method that uses the sanitised description, then return it as `error` from `ReadOAuthError` (drop the parallel `error` return). Removes the duplicate templating at the two call sites and prevents future callers from formatting the raw description directly.

## Type: `tokenmanager.TokenRequest`

File: `/Users/alex/workspace/auth-go/tokenmanager/tokenmanager.go:264-281`

### Invariants Identified
- `Resource` non-empty (checked at `:303-305`).
- `Resource` is an origin URL (no path/query/fragment, no userinfo, https unless `AllowInsecureHTTP`). Enforced via `oauthhttp.ValidateOriginURL` at `:306-309`.

### Ratings
- **Encapsulation**: 5/10. Plain struct with public fields, no methods. That's typical Go and fine for a parameter object, but the field doing "raw input → normalised output" double duty (per the prompt's question) muddies the contract.
- **Invariant Expression**: 4/10. The field doc says "Resource is the origin where the bearer will be presented. Required." It doesn't say "Resource must be an https origin URL with no path/query/fragment, or http if AllowInsecureHTTP." A caller reading the type definition has to read three files (`tokenmanager.go`, `oauthhttp/origin.go`, and the `Config.AllowInsecureHTTP` docs) to know what's actually accepted. This is a doc-only invariant.
- **Invariant Usefulness**: 9/10. The normalisation + validation prevents real bugs: trailing-slash differences shadowing cache entries, accidentally sending bearers over HTTP to a non-loopback host, accepting `https://api.example.com/path/with/query?bad=true` as a "resource".
- **Invariant Enforcement**: 8/10. Validation happens at the entry of `Token` (`:306-309`), before any side effects. Good. The new `multi-slash` guard (commit `dcfefbe`) in `ValidateOriginURL` is the right place for it. Caching key uses the normalised value (`:336, :352`), so the post-normalisation form is what flows through.

### Strengths
- Single validation point in `Token`; no scattered re-validation.
- The "default Audience to normalised Resource" rule at `:349-351` is well-commented and explains the zitadel quirk.

### Concerns
1. **`resolved.Resource = normResource` (line 336) is the exact pattern flagged in the prompt.** `TokenRequest` is being used as both the caller's input AND the manager's internal "resolved" representation. `resolve` (`:366-374`) takes a `TokenRequest` and returns a `TokenRequest`; the type doesn't distinguish "what the caller asked for" from "what we're going to send". This is the classic anemic-data-class-doing-two-jobs anti-pattern from the rubric.
   - The mutation works correctly today, but a reader has to trace `req` vs `resolved` vs `normResource` to understand which copy holds which form. The diff shows `normResource` is now derived earlier (`:306`) than before — which is correct for the security check — but means the variable lives across more lines, with `req.Resource` and `resolved.Resource` and `normResource` all in scope simultaneously between `:306` and `:357`.

2. **Zero value is invalid** (Resource required). Same Go idiom as `Config`. Acceptable but worth noting.

3. **No constructor.** `TokenRequest{Resource: "..."}` is the only construction path. For a public type with this many invariants, a `NewTokenRequest(resource string) (TokenRequest, error)` would let callers fail-fast at construction rather than at `Token()`.

### Recommended Improvements
- **Introduce a separate `resolvedRequest` (unexported) for the internal normalised form.** Keep `TokenRequest` as the user-facing input type. Roughly:
  ```go
  type resolvedRequest struct {
      Resource           string // normalised
      Audience           string
      RequestedTokenType string
      Scope              string
  }
  ```
  Then `resolve(req TokenRequest, normResource string) resolvedRequest` returns a distinct type. The line-336 mutation goes away. `makeCacheKey` and `runExchange` take a `resolvedRequest`, making "this Resource is normalised" a type-level guarantee. Modest code cost; eliminates the "same field, two meanings" confusion.
- Alternative low-cost option: rename the internal variable from `resolved` to something like `wireReq` or `outboundReq` to make the dual-purpose explicit, and reassign all four fields rather than just `Resource` so it's clear this is the outbound form. Cheaper than a new type; less guarantee.
- Field doc for `Resource` should mention the validation contract directly — at minimum: "Must be an https origin URL (or http if Config.AllowInsecureHTTP). Path, query, fragment, and userinfo are rejected. Normalised via RFC 3986 §6.2.2 before use."

# Error Handling Audit: PR #5 (soph/more-improvements-fixes)

Overall: error handling on this branch is meaningfully better than what it replaces — `oauthhttp.ReadOAuthError` and `ReadAndDecodeJSON` consolidate previously scattered, partly silent code (e.g., the old `io.ReadAll` with `_` discard in `sts/sts.go`). I have **no critical** silent-failure findings; the remaining issues are bounded leaks of unsanitised AS content into the user's terminal, one piece of implicit `nil` contract, and a couple of UX/diagnostic gaps.

## Important

**1. `apiErr.Error` echoed unsanitised into terminal-bound errors** — `sts/sts.go:434, 436` and `deviceflow/deviceflow.go:488, 490` (via `errCodeToSentinel` → `deviceflow/deviceflow.go:232`)
- What's silent: `apiErr.ErrorDescription` is run through `oauthhttp.SanitizeDescription`, but `apiErr.Error` (the OAuth error *code*) is interpolated raw with `%s`. RFC 6749 §4.1.2.1 limits the code to ASCII-printable, but the AS is the only enforcer — a buggy/hostile server can ship `"error":"invalid_grant\u001b[31m"` and the bytes reach the terminal.
- Why it matters: `ReadOAuthError` was added precisely to keep terminal-toxic content out of error chains; this is the matching hole on the sibling field. CSI (U+009B) in particular slips through any "low-byte" filter and is interpreted by 8-bit-aware terminals.
- Fix: sanitise the code the same way: `code := oauthhttp.SanitizeDescription(apiErr.Error)`. For `deviceflow.errCodeToSentinel`'s `default:` arm at `deviceflow/deviceflow.go:232`, sanitise `code` before interpolation: `fmt.Errorf("oauth error: %s", oauthhttp.SanitizeDescription(code))`.

**2. `validateVerificationURI` does not block C1 control bytes** — `deviceflow/deviceflow.go:315-319`
- What's silent: The loop rejects `r &lt; 0x20 || r == 0x7f` but lets U+0080–U+009F through. U+009B (CSI) is the 8-bit-equivalent of `ESC [`; printing the raw verification_uri opens ANSI on terminals that decode C1. `oauthhttp.SanitizeDescription` already blocks this same range (`internal/oauthhttp/sanitize.go:46-47`); the verification URI is held to a weaker bar even though it gets echoed verbatim to the user and (per the doc) is exactly the surface the validation is meant to harden against terminal-glance attacks.
- Why it matters: the function's whole stated purpose is "must not contain control characters … that could break terminal output." A 600-byte `https://example.com/path&lt;U+009B&gt;[31m…` payload passes today.
- Fix: extend the range check to include C1 controls: `if r &lt; 0x20 || r == 0x7f || (r &gt;= 0x80 &amp;&amp; r &lt;= 0x9f)`. Or factor out a shared `isTerminalControlRune` from `sanitize.go`.

**3. `ReadOAuthError`'s nil-vs-non-nil return contract is implicit** — `internal/oauthhttp/error.go:25-46`
- What's silent (today): nothing — by construction the only path that returns `(*OAuthErrorResponse, nil)` is `apiErr.Error != ""`, so callers in `sts.readAPIError`/`deviceflow.readAPIError` are safe to dereference. But the function signature offers no static guarantee, the godoc doesn't state it, and a future refactor that re-enables (say) "return apiErr even with empty `error` field" would yield `(nil-deref, nil)` panics at the call sites.
- Why it matters: caller code at `sts/sts.go:433-436` and `deviceflow/deviceflow.go:486-490` dereferences `apiErr.Error` immediately after the `parseErr != nil` early-return. A single edit to `ReadOAuthError`'s middle branch could turn this into a nil-pointer panic deep inside an error path that's exercised only on AS failures.
- Fix: either (a) document the invariant in `ReadOAuthError`'s godoc ("returns `(*OAuthErrorResponse, nil)` only when `Error` is non-empty"), or (b) defend at the call sites with `if apiErr == nil { return fmt.Errorf("token exchange: status %d", resp.StatusCode) }`. (a) is cheaper.

**4. `Extra["client_id"]` with multiple entries: only the first divergent value surfaces** — `sts/sts.go:179-183`
- What's silent: `validateClientIDConsistency` loops over `extra["client_id"]` and returns on the first mismatch. If the slice has `["a", "b"]` (already weird) and the typed ClientID is `"a"`, only `"a"` is checked; `"b"` is silently accepted. More importantly, when `Extra["client_id"] = ["a", "b"]` with no typed ClientID, the function returns nil because of the `id == ""` short-circuit at line 176, even though `Extra` itself is internally inconsistent — and `buildForm` will then write both values to the wire.
- Why it matters: zitadel and other servers parsing `r.PostFormValue("client_id")` take only the first; servers parsing `r.PostForm["client_id"]` see both. A misconfigured caller could end up sending two different client IDs on the same request and have it succeed against one server and fail opaquely against another.
- Fix: reject `len(extra["client_id"]) &gt; 1` outright (single-value invariant), and run the typed-vs-Extra mismatch check on the canonical first element only after that guard. Both branches are misconfiguration, not user error.

## Nits

**5. Inconsistent `strict` JSON decoding between device-auth start and poll** — `deviceflow/deviceflow.go:263` uses `strict=true`, `deviceflow/deviceflow.go:404` and `sts/sts.go:311` use `strict=false`.
